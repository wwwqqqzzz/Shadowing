# Shadowing 微信小程序项目方案

## 项目定位

这个项目不是一个“语言播放器”，也不是一个“大而全 AI 学习平台”。

它更接近于一个：

“围绕 shadowing（影子跟读）训练流程构建的轻量语言训练系统”。

核心目标不是做内容生产，也不是做复杂 AI，而是：

让用户能够低成本、低门槛地开始跟读训练，并且在训练过程中获得明确反馈。

整个产品最重要的不是功能数量，而是训练闭环是否顺滑。

也就是说：

用户打开小程序。
选择素材。
开始播放。
系统自动暂停。
用户开口跟读。
系统进入下一句。

这个过程是否足够自然、低摩擦。

这是整个产品最核心的体验。

如果这个流程卡顿、复杂、等待时间长，那么即使后面接入再高级的 AI，也无法形成真正的使用习惯。

因此，项目第一阶段的目标不是“智能”，而是“稳定、顺滑、完整”。

---

# 为什么适合先做 Shadowing

相比 AI 对话类语言产品，shadowing 有一个现实优势。

它是结构化训练。

用户输入、系统逻辑、交互路径都相对固定。

系统只需要解决：

* 音频播放
* 字幕同步
* 自动暂停
* 跟读录音
* 简单反馈
* 训练记录

而不是开放式对话。

因此它更适合个人开发者独立完成。

另外，shadowing 天然适合碎片化。

用户不需要专门安排完整学习时间。

地铁、通勤、午休都可以练。

这类产品真正的竞争力并不在 AI，而在于：

“用户是否愿意每天打开”。

所以整个系统设计，应当围绕“持续训练”展开，而不是围绕“炫技功能”展开。

---

# 第一阶段应该做成什么样

项目第一阶段不应该追求完整。

而应该追求：

“像一个真正能上线的产品”。

也就是说：

* 有完整训练流程
* 有内容体系
* 有用户系统
* 有数据记录
* 有后台管理
* 有稳定播放体验

但不要追求：

* AI 发音专家
* 社交裂变
* 社区
* 用户上传平台
* 海量内容生态

这些都会把项目复杂度迅速拉高。

第一阶段建议只做：

英语。

因为：

* 英语公开素材最多
* 字幕资源最多
* 语音识别成熟度最高
* 用户习惯最成熟

日语和韩语可以后续扩展。

但架构设计阶段，要提前预留多语言能力。

---

# 产品结构设计

整个系统建议拆成四部分。

分别是：

首页。
素材列表。
跟读训练页。
个人中心。

其中真正核心的是“跟读训练页”。

因为用户最终停留时间最长的地方一定是这里。

---

## 首页

首页不需要复杂。

不要做资讯流。
不要做推荐算法。
不要做花里胡哨的运营模块。

首页只需要承担两个任务：

第一，让用户快速进入训练。

第二，让用户知道今天该练什么。

因此首页建议保留：

* 今日推荐素材
* 最近练习
* 分类入口
* 难度入口

整个页面重点是：

减少思考成本。

用户最好在 10 秒内进入播放页。

---

## 素材列表

素材系统建议采用：

“固定精选内容”

而不是：

“开放式海量内容”。

因为个人开发阶段根本没有能力做内容运营。

所以素材应当偏少，但质量稳定。

每个素材至少需要：

标题。
语言。
难度。
时长。
字幕。
音频。
封面。

难度建议只做三级：

初级。
中级。
高级。

不要做复杂 CEFR。

没有必要。

第一阶段素材来源建议全部来自公开资源。

比如：

YouTube 字幕。
VOA。
BBC Learning English。
TED-Ed。

系统只需要：

获取字幕。
切句。
同步时间轴。

不要自己生产内容。

---

## 跟读训练页

这是整个系统最核心的部分。

所有开发重点都应该放在这里。

第一阶段必须具备的功能包括：

字幕逐句高亮。
播放控制。
自动暂停。
变速播放。
单句循环。
录音。
录音回放。
下一句切换。

其中最重要的是：

自动暂停。

因为 shadowing 的本质不是“看字幕”，而是“强迫开口”。

系统必须主动制造跟读节奏。

理想流程应该是：

播放一句。
自动暂停。
用户开口。
自动进入下一句。

整个流程尽量减少点击。

不要让用户频繁操作按钮。

因为语言训练本身已经很累。

交互越复杂，越容易放弃。

---

## 个人中心

个人中心第一阶段不用复杂。

只保留：

训练记录。
累计时长。
最近练习。
收藏内容。

重点是让用户知道：

自己不是“看了内容”。

而是“真的练了”。

因为训练感会直接影响留存。

---

# 关于 AI 评分

AI 评分可以做。

但不应该第一阶段就做“专业发音评分”。

因为真正的发音评分非常复杂。

不仅仅是语音识别。

还涉及：

音素对齐。
重音。
连读。
韵律。
语调。

这是完整语音工程领域。

如果一开始直接进入这个方向，很容易把项目拖进长期开发泥潭。

因此更现实的路线是：

先做“轻反馈”。

例如：

用户读完一句。
系统做 ASR。

然后把识别结果与原句进行比较。

如果：

单词缺失。
发音错误导致识别失败。
语速异常。

则给出简单提示。

这种方式虽然不是真正意义上的“专业口语评分”，但已经足够形成训练反馈。

而且工程复杂度低很多。

真正专业评分建议放在第二阶段之后。

---

# 技术架构建议

整个项目建议采用：

“小程序前端 + 独立后端服务”的结构。

不要重度依赖微信云开发。

因为后面一定会碰到：

音频处理。
转码。
ASR。
任务队列。
素材处理。

这些都更适合独立服务。

建议整体结构如下：

微信小程序。

负责：

播放。
录音。
界面。
训练流程。

后端 API 服务。

负责：

用户。
素材。
记录。
权限。
数据。

音频处理服务。

负责：

切句。
转码。
字幕同步。
音频处理。

AI 服务。

负责：

ASR。
文本比较。
后续发音分析。

这样后面即使功能升级，也不会推翻整体结构。

---

# 技术选型建议

前端建议使用原生微信小程序。

因为：

你当前目标不是多端。

而是：

稳定。
顺滑。
快速上线。

跨端框架虽然方便，但音频相关 API 经常出现兼容问题。

尤其是：

录音。
播放。
后台状态。
蓝牙耳机。

这些在小程序里本身就已经很麻烦。

因此第一阶段建议优先稳定。

---

后端建议使用 NestJS。

原因包括：

结构清晰。
适合模块化。
适合后台系统。
适合权限体系。

数据库建议使用 PostgreSQL。

因为后面会涉及大量关系结构。

例如：

用户。
素材。
句子。
训练记录。
收藏。

这些都更适合关系型数据库。

不建议一开始使用 MongoDB。

---

# 为什么一定要做后台管理系统

后台不是“可选功能”。

而是整个系统后期稳定运行的基础。

因为你本质上是在做：

内容驱动型训练平台。

而不是单机工具。

后面一定需要管理：

素材。
字幕。
句子。
用户。
训练数据。

如果没有后台。

后面所有内容维护都会非常痛苦。

尤其是：

字幕错误。
素材下架。
难度调整。
用户反馈。

都会变成灾难。

---

# 后台应该具备什么

第一阶段后台不需要复杂。

但至少需要：

素材管理。
句子管理。
用户管理。
训练记录查看。

其中素材管理最重要。

后台应该支持：

上传素材。
导入字幕。
自动切句。
调整句子。
修改时间轴。
设置难度。
发布内容。

因为 shadowing 的核心，其实是：

“句子级时间轴”。

如果字幕切得不好。

训练体验会非常差。

因此后台必须允许人工调整。

---

# 数据结构建议

系统里最核心的数据不是用户。

而是“句子”。

因为整个训练过程其实围绕句子展开。

因此建议数据结构以：

素材 -> 句子 -> 训练记录

为核心。

例如：

一个素材包含很多句子。

每个句子拥有：

开始时间。
结束时间。
文本。
翻译。

用户训练记录则绑定句子。

这样后期才能实现：

错题。
重复训练。
难点分析。

如果一开始不这么设计。

后面重构会非常痛苦。

---

# 开发方式：Vibe Coding 而不是传统软件工程

这个项目很适合使用 Vibe Coding 方式推进。

因为：

它的核心复杂度不在算法。

而在：

交互流程。
页面联动。
播放节奏。
训练体验。

所以相比“先设计完整架构再写代码”，更适合采用：

“快速生成 -> 立即运行 -> 实际体验 -> 反复调整”

的方式。

但这里有个非常关键的问题。

很多人理解的 Vibe Coding 是：

“想到什么做什么”。

最后项目会迅速失控。

真正有效的 Vibe Coding 必须建立在：

“明确系统边界”

之上。

也就是说：

虽然开发过程是快速生成。

但产品结构必须稳定。

因此整个项目应该拆成：

固定核心。

和：

可快速迭代模块。

---

# 系统核心边界

整个系统真正不可推翻的核心只有几个。

用户。
素材。
句子。
训练记录。

无论后面怎么升级 AI。

怎么改 UI。

怎么换播放器。

整个系统最终都还是围绕：

“用户对句子的训练”

展开。

所以项目架构设计一定要围绕这几个实体。

否则后期一定重构。

---

# 整个项目的完整阶段规划

这个项目建议分成六个阶段。

每个阶段都必须：

可运行。
可体验。
可上线。

不要等“全部做完再上线”。

那种项目最后大概率烂尾。

---

# Phase 0：原型验证阶段

这一阶段不要考虑 AI。

不要考虑数据库。

不要考虑后台。

只做一件事：

验证 shadowing 流程是否舒服。

这个阶段甚至可以不接后端。

直接本地 mock 数据。

目标是：

做出一个可以播放字幕、自动暂停、录音的页面。

你需要重点验证：

* 自动暂停节奏
* 字幕高亮
* 录音按钮交互
* 单句循环体验
* 变速播放体验

这一阶段最重要的是：

体验感。

不是代码质量。

---

## Phase 0 推荐 Prompt

适合直接丢给 Cursor / Claude / GPT。

“帮我生成一个微信小程序 shadowing 跟读页面。页面需要支持字幕逐句高亮、音频播放、自动暂停、单句循环、录音按钮、变速播放。整体风格偏极简深色 UI。重点优化移动端训练体验。”

# Phase 2：素材处理系统

这一阶段重点不是用户。

而是内容处理能力。

因为 shadowing 的核心其实是：

“句子级时间轴”。

这一阶段需要解决：

字幕导入。
自动切句。
时间轴修正。
音频切分。

你后面所有训练体验，都依赖这个阶段。

如果句子切得不好。

用户会读得非常痛苦。

---

ffmpeg。

## Phase 2 推荐 Prompt

“帮我设计一个字幕处理系统。输入为 srt/vtt 字幕与音频，输出为句子级时间轴数据。系统需要支持自动切句、时间轴修正、导出 JSON 结构，适用于 shadowing 跟读训练。”

---

# Phase 3：轻量 AI 反馈系统

这一阶段才开始引入 AI。

但不要碰复杂发音工程。

第一阶段 AI 目标应该是：

“用户知道自己哪里读错了”。

而不是：

“专业语言考试级评分”。

---

## 推荐实现

用户录音。
上传。
ASR 转文本。
原文比较。
返回错误词。

例如：

原句：

How are you doing today

识别结果：

How you doing today

系统提示：

缺失 are。

这已经能形成训练反馈。

---

## Phase 3 推荐 Prompt

“帮我设计一个 shadowing 跟读轻量 AI 反馈系统。用户录音后，系统通过 ASR 转文字，再与原句比较，返回错误单词、高频漏读词、流利度信息。请设计接口结构与数据流。”

---

# Phase 4：专业发音分析

这是最难的一阶段。

也是最容易陷进去的一阶段。

韵律。
连读。
intonation。

这已经接近完整语音工程。

这一阶段建议优先接第三方。

不要自己造轮子。

# 真正需要警惕的问题

这个项目最大的风险不是技术。

而是：

“不断加功能”。

因为这种产品特别容易进入：

背单词。
AI 对话。
社交。
打卡。
排行榜。
社区。

最后变成一个巨大缝合怪。

但 shadowing 真正核心其实只有一句话：

“让用户持续开口。”

所以你后面每次想加功能时。

都应该问自己：

这个功能是否真的能提升开口训练。

如果不能。

大概率不应该现在做。

---

# 第一阶段真正应该完成的目标

第一阶段不要追求用户量。

也不要追求商业化。

真正应该完成的是：

“一个完整、成熟、能上线的小系统”。

具体来说，就是：

用户可以注册。
可以进入素材。
可以跟读。
可以录音。
可以查看历史。
后台可以管理内容。
系统可以稳定运行。

做到这一步，其实已经是一个完整项目。

而不是 Demo。

---

# 第一阶段最重要的判断标准

不是：

AI 是否先进。

而是：

用户是否真的愿意连续练 10 分钟。

因为 shadowing 产品真正的核心竞争力不是功能数量。

而是训练节奏。

如果训练节奏舒服。

即使没有 AI，用户也会持续使用。

如果训练节奏糟糕。

再高级的 AI 也救不了。

因此整个项目开发过程中，应该始终围绕一个问题：

“用户能否自然地持续开口”。
